# 0. 开始之前

欢迎加入 AMLL TTML 审核团，感谢你为本项目做出的额外贡献！

# 1. 你需要做什么？

- 按照[内容指南](instruction.md)审核大家的投稿；
- 提出修改建议；
- 关注群聊中是否出现不合规内容并及时撤回/删除；
- 在**有额外精力**的条件下对仓库进行日常查看和维护。

# 2. 具体怎么做？

- 内容审核部分：

  - 对于新来的贡献者：

    1. 请尽量查看 TA 投稿中的每一行歌词的：

       - 时间轴是否准确（100 ms 以内，不存在肉眼可见误差）；

       - 歌词内容是否符合规范；

       - 排版是否合理；

       - 以及其他你认为可能不合理的地方，

    1. 并检查PR页面上的：

       - 元数据填写是否正确；

       - 文件内容是否正确；

       - （如果是修改投稿）是否有关联到上一份投稿。
       
         > 如果投稿者留言表示“未修改时间轴”，你可以点开看看 TA 是不是真的没改，~~又或者可以无条件相信 TA ，把问题交给后来者（bushi）~~。

  - 对于曾经投过稿的贡献者，你可以根据对 TA 前次投稿的评估情况（如准确度、内容质量等）进行选择性地抽样审核以提高审核效率。如果你确实有空，更推荐逐句看完。 但请注意：**这有概率引起一些细节上的争议，所以请量力而行。**
  
    > 除此之外，对于具有出处或 IP 关联的曲目，在时间条件允许的情况下请尽可能去查证投稿与原始内容是否吻合、投稿对原始内容的表达和处理是否得当等。

> [!CAUTION]
>
> 审核人员原则上不能审核自己的投稿，我们不能既当裁判又当运动员。推荐做法是将这份投稿交予其他与这份投稿无关的审核，由 TA 进行批阅。
>
> 例如：
>
> 审核 A 提交了一份投稿，由审核 B 批阅并通过，然后 B 再次提交对这份投稿的修改，那么审核 A 与 B 将不能参与对这份投稿的批阅。这是**具有传染性**的，但传递 3 次以上即可视为一份新投稿。即使是这样，最近一次参与批阅这份投稿的审核也仍然不能再次参与批阅这份投稿。

- 结果反馈部分：
  - 如果投稿者为审核团内部人员，可无需在 PR 页面上填写审核结果或修改意见，在群内或私聊告知对方即可。
  - 如果投稿者非审核团内部人员，无论是否通过，你都应该在 PR 页面上留下一点什么，哪怕只是感谢 TA 对本项目的贡献。
  - 反馈结果的格式不做要求，但是请保证：
    - 一次性指出当前投稿中存在的所有问题；
    - 指出的问题点是清晰的、可供修改的：
      - ✔正确示例：XX 行的 XXX 单词存在 XXX 问题。
      - ✔正确示例：XX 行不应是 XX 效果，需要改为 XX。
      - ✔正确示例：元数据存在 XX 问题。
      - ❌错误示例：XX 行有问题。
    - **切记不要以群内消息记录代替 PR 留言**，因为即时群组通讯工具不适合用于此类有目标性的、需要一次性表达的场合；
    - 请尽可能使用书面化、完整、清晰的语言进行回复。

# 3. 遇见错误怎么处理？

1. 发现错误后及时打开歌词源文件进行核查疑似出错的部分；
1. 如果能够确认出现错误，则可以直接向歌词作者回复；
1. 如果无法自行确认，则可以与其他审核进行讨论，根据讨论结果回复歌词作者；
1. 如果是词库本身的缺陷导致歌词有瑕疵，则需要暂时将对应提交打上「慢速审核」标签，并及时向歌词作者说明原因。之后发起「词库议题」Issue 说明词库需要修改的部分，等待词库修改后及时通知歌词作者再次提交并重新审核。

> [!CAUTION]
>
> 严禁吹毛求疵，将一些在合理范围内无关紧要的（也可能算不上瑕疵的）修改点也作为修改要求退回给投稿者，避免对双方造成时间和其他成本上的浪费。
>
> 例如：
> - 偏差了 ±十几毫秒 但并没有覆盖到其他单词的时间轴；
> - 为了上下文连贯等特殊场景故意拖长的 **行** 时间轴；
> - 其他不影响时间轴准确度的修改点。

# 4. 作为管理者的行为规范

1. 严禁在公开场合泄露贡献者的非公开联系方式、项目团队内部群聊中不适合公开的（或容易引起他人产生误解的）内容或未经允许的未发布草案等；
2. 作为社区的专业服务者，而非“管理者”，务必避免以高人一等的姿态与他人交流。如果被询问了不属于你的知识范围的内容，可以请其他具有相关知识/技能的审核员或社区内对此有了解的用户进行解答，或引导 TA 自行搜索阅读相关文档，不要使用模糊的态度回复提问者；
3. 在社区（如 QQ 群、Discord、GitHub Discussions）讨论时，明确区分“个人观点”与“项目官方立场”；

# 5. 活跃度与责任
_此部分自 第二批审核招募 截止之时起生效，对此批次（含）起加入的成员生效_

1. 通过项目维护方式（如维护 AMLL 主仓库、 AMLL TTML Tool 、镜像站等）加入的成员可不必遵守此限制；
2. 为了保证项目的稳健运行和贡献者的良好体验，审核员需维持基本的活跃度。你需要保证每周至少能处理 4 份投稿（`Approve` / `Request_changes` / `Merge` 操作视为有效计数 ~，可在 AMLL TTML Tool - test 中查看~ ），若连续 2 周未达标且未说明原因，我们可能考虑将你标记为非活跃成员，并视情况将你请出审核团队，但我们依然欢迎你在下一次招募中重新申请加入；
3. 若因学业、工作或个人原因短期内（超过 1 周）无法处理审核工作，请在群内说明。
